# Next Resume PRD

版本：v0.1  
日期：2026-05-07  
项目：`next-resume`  
技术边界：只使用 Next.js 全栈能力，不引入独立后端框架。

## 1. 产品概述

`next-resume` 是一个个人简历全栈应用。它以公开简历页面为核心，使用 Next.js App Router 渲染页面，使用 Server Components 读取 Neon/PostgreSQL 数据，使用 Server Actions 处理联系表单、留言等写入操作。

产品第一阶段目标不是做复杂招聘平台，而是做一个可上线、可维护、可扩展的个人简历站点：页面清晰、响应式良好、SEO 友好、支持打印/PDF，并能接收访客联系消息。

## 2. 背景与问题

传统简历通常是 PDF 或静态页面，更新成本高，项目经历、技能和联系方式难以结构化维护。本项目希望通过 Next.js 全栈能力，把简历内容结构化存储到数据库，并用服务端渲染输出高质量页面。

需要解决的问题：

- 简历内容可以结构化维护，而不是全部硬编码在 JSX 中。
- 公开页面加载快、SEO 友好，并适合移动端查看。
- 访客可以通过表单留下联系消息。
- 技术栈保持简单：Next.js + Neon/PostgreSQL，不拆独立后端。

## 3. 用户角色

### 3.1 简历所有者

- 维护个人信息、技能、经历、项目、教育和证书。
- 查看联系消息。
- 需要稳定、低维护成本的部署方式。

### 3.2 访客 / 招聘方

- 快速了解候选人背景、能力、项目和联系方式。
- 在手机或桌面端阅读简历。
- 下载或打印简历。
- 通过联系表单发送消息。

## 4. 产品目标

- 建立一个专业、可读、响应式的中文简历页面。
- 简历主要内容来自 Neon/PostgreSQL。
- 使用 Next.js Server Components 做服务端数据读取。
- 使用 Next.js Server Actions 做表单提交和数据写入。
- 支持 SEO metadata、Open Graph 基础信息和打印样式。
- 保持依赖少、结构清楚，便于后续迭代后台管理能力。

## 5. 非目标

- 不做多用户招聘平台。
- 不做在线支付、订阅、复杂权限系统。
- 不引入 Express、NestJS、Koa、FastAPI 等独立后端。
- 不在第一阶段实现复杂 CMS 编辑器。
- 不把简历页面做成营销落地页或过度动画页面。

## 6. MVP 范围

### 6.1 公开简历页 `/`

展示：

- 姓名、职位标题、所在地、联系方式、个人链接。
- 个人简介。
- 技能分组。
- 工作经历。
- 项目经历。
- 教育经历。
- 证书或奖项。
- 联系入口。

要求：

- 首屏直接展示简历核心信息。
- 桌面端和移动端布局都必须清晰。
- 支持浏览器打印，打印时隐藏不必要的交互控件。

### 6.2 项目详情页 `/projects/[slug]`

展示单个项目的详细描述、技术栈、亮点、链接和时间范围。

如果 MVP 需要压缩范围，可以先不做独立详情页，在首页展示完整项目卡片。

### 6.3 联系表单

字段：

- 姓名。
- 邮箱。
- 公司或来源，可选。
- 主题，可选。
- 消息内容。

处理方式：

- 使用 Server Action 接收 `FormData`。
- 服务端校验字段类型、长度和必填项。
- 写入 `contact_messages` 表。
- 返回成功或失败状态。

### 6.4 简历内容数据源

第一阶段内容维护方式：

- 通过 Neon SQL Editor 插入或更新数据。
- 应用只负责读取和展示。

后续可扩展：

- `/admin` 管理页面。
- 受保护的 Server Actions。
- 简历内容编辑表单。

## 7. 页面与路由

| 路由 | 类型 | 用途 |
|---|---|---|
| `/` | Server Component | 公开简历首页 |
| `/projects/[slug]` | Server Component | 项目详情页，可选 |
| `/api/health` | Route Handler，可选 | 部署健康检查 |
| `/admin` | Server Component + Server Actions，后续 | 简历管理后台 |

Route Handler 只用于外部系统调用、健康检查或 webhook。页面内部表单优先使用 Server Actions。

## 8. 功能需求

### FR-01 读取公开简历

系统应从 `resumes` 表读取已发布简历，并按排序字段读取技能、经历、项目、教育和证书。

验收标准：

- 只展示 `is_published = true` 且相关条目 `is_visible = true` 的内容。
- 内容为空的 section 不渲染空标题。
- 数据库异常时页面应给出稳定错误状态或降级内容。

### FR-02 展示个人信息

系统应展示姓名、标题、简介、所在地、邮箱、电话、网站和社交链接。

验收标准：

- 邮箱和外链可点击。
- 外链使用安全属性。
- 移动端不发生文字重叠或横向溢出。

### FR-03 展示技能

系统应按技能分组展示技能项。

验收标准：

- 支持分组排序。
- 支持技能项排序。
- 技能等级字段可选，不强制展示。

### FR-04 展示经历

系统应展示公司、职位、时间范围、地点、简介和亮点列表。

验收标准：

- 当前任职支持 `is_current = true`。
- 时间范围格式一致。
- 亮点列表可为空。

### FR-05 展示项目

系统应展示项目名、角色、描述、技术栈、亮点、演示链接和源码链接。

验收标准：

- 支持精选项目排序。
- 支持通过 `slug` 访问详情页。
- 不可见项目不展示。

### FR-06 联系消息提交

访客可提交联系消息。

验收标准：

- 服务端校验必填字段。
- 成功写入 `contact_messages`。
- 不把数据库错误细节暴露给前端。
- 防止空消息、超长消息和非法邮箱。

### FR-07 打印 / PDF 友好

简历页面应支持浏览器打印。

验收标准：

- 打印时隐藏联系表单、按钮、动画和非必要装饰。
- 主要内容黑白打印可读。
- 页面断点尽量合理。

## 9. 数据模型

核心表：

- `resumes`：简历主档。
- `resume_links`：个人链接。
- `skill_groups`：技能分组。
- `skill_items`：技能项。
- `work_experiences`：工作经历。
- `projects`：项目经历。
- `education`：教育经历。
- `certifications`：证书、奖项或认证。
- `contact_messages`：访客联系消息。
- `comments`：兼容当前初始表单的留言表。

建表脚本见：`database/neon-schema.sql`。

## 10. 技术方案

### 10.1 前端

- 使用 Next.js App Router。
- 页面默认 Server Component。
- 只在需要交互状态时拆 Client Component。
- 前端 UI 优先使用项目已安装并配置的 shadcn/ui；按钮、表单、输入框、卡片、徽标、弹窗、分隔线等基础界面组件优先从 `@/components/ui` 组合。
- 使用 Tailwind CSS v4 做样式。
- 业务级简历模块可以在 shadcn/ui 基础上组合封装，但不要重复手写已有基础组件。
- 保持简历信息层级清楚，不做过度装饰。

### 10.2 服务端

- 数据读取：页面和组件不直接访问数据库；需要业务数据时调用 service 或专用 controller。
- 数据写入：Server Actions 必须放在 `lib/controllers/**`，作为 controller 只负责解析请求、鉴权入口、调用 service 和返回/跳转。
- 外部 API：Route Handlers。
- 不引入独立后端服务。

### 10.3 数据访问

- 使用 `@neondatabase/serverless`。
- 数据库客户端不得在模块顶层直接初始化。
- 推荐封装懒加载 getter，例如 `getSql()`。
- 所有数据库能力强制使用三层架构：
  - `controller`：固定放在 `lib/controllers/**`，所有 Server Actions、Route Handler 委托函数和请求入口控制逻辑都必须放在这里，处理 `FormData`、headers、cookies、redirect，不允许导入 `getSql()` 或编写 SQL。
  - `service`：`lib/services/**`，实现业务规则、鉴权授权语义、审计编排和跨 DAO 流程，不处理 UI 状态和请求对象。
  - `dao`：`lib/dao/**`，唯一允许调用 `getSql()` 和编写 SQL 的业务层，负责参数化查询和数据库行映射。
  - `app/**` 目录只放页面、布局和 UI 组合，不新建 `actions.ts` 承载 controller。
- Neon SQL 调用规则：
  - 标签模板：`sql\`SELECT ${value}\``。
  - `$1` 参数：`sql.query('SELECT $1', [value])`。

### 10.4 缓存与刷新

- 公开简历内容以读取为主，可以使用 Next.js 缓存能力。
- 写入联系消息后不需要刷新简历内容。
- 如果后续加入后台编辑，更新简历内容后应按 Next.js 16 文档使用对应的 revalidation API。

## 11. 安全与隐私

- `.env*` 不提交，不打印，不在日志中暴露。
- `DATABASE_URL` 只在服务端使用。
- Server Actions 必须做输入校验。
- 联系消息不保存原始 IP，最多保存不可逆 hash 或来源信息。
- 后续管理后台必须加入鉴权和授权检查。
- 对外错误提示应简洁，不暴露 SQL、连接串或堆栈细节。

## 12. 性能要求

- 首页主要内容应由服务端渲染输出。
- 避免把整份简历数据传给不必要的 Client Component。
- 图片需要设置尺寸、alt 文本和合适的优化方式。
- 首屏不依赖大型客户端 JS。
- 生产构建必须通过 `pnpm build`。

## 13. 可访问性要求

- 标题层级合理。
- 链接和按钮可键盘访问。
- 表单字段有可访问标签。
- 颜色对比度满足基本阅读需求。
- 不把关键信息只放在 hover 或动画状态里。

## 14. 验收标准

MVP 完成时应满足：

- `pnpm lint` 通过。
- `pnpm build` 通过，除非外部字体网络问题导致失败，需明确说明。
- Neon 建表 SQL 可执行。
- 首页可从数据库读取并展示简历内容。
- 联系表单可写入数据库。
- 桌面端和移动端视觉检查通过。
- 打印预览主要内容可读。
